# ADR-0001 B2B 受付モードと SNS モードの分離

## ステータス: Accepted (2026-07-10) / Revised (2026-07-10)

## 背景

- 現行 ja ウェルカムは「おっす！…やけん」で、士業・建設・製造の問い合わせ受付にはカジュアルすぎる。
- Contact フォーム代用として、構造化リード・機密入力禁止・spam 対策が必須（要件定義 AI Contact Reception）。
- 会話が長すぎると離脱する（目標: 通常 3〜6 往復、必須最大約 7 問、2〜5 分）。

## 決定

### モード

| モード | 用途 | トーン |
|---|---|---|
| **intake（本線）** | cor-jp.com お問い合わせ代用 | 敬語・簡潔・業務的 |
| **ambassador（副）** | SNS / イベント / デモ | 福岡弁・キャラ寄り可 |

- 本番 Contact（URL / 埋め込み）は **intake 固定**。

### 初期文案（intake）

> こんにちは。Cor.株式会社のAI受付 Cloudiaです。数問でご相談内容を整理し、担当者へ引き継ぎます。顧客名、個人情報、契約書本文などの機密情報は入力しないでください。まず、ご相談の目的を選んでください。

### 初期選択肢（intent）— corsweb ADR-0010 正本

| key | 表示（ja） |
|---|---|
| `confidential-ai-assessment` | 機密データAI活用診断 |
| `local-llm-poc` | ローカルLLM / AI基盤PoC |
| `grift-team-beta` | Grift Team Beta |
| `grift-paid-trial` | Grift Paid Trial |
| `estimate-audit` | Estimate Audit |
| `press-speaking-other` | 取材・登壇・その他 |

- 未知キーは未選択にフォールバック（ログ可）。
- API に載せる key は上記と同一。

### 会話フロー（要件 §6.1 準拠）

1. 目的（intent）
2. 業種・役割・課題（顧客名・契約本文は促さない）
3. データ種類と機密度（具体本文は入力させない）
4. 進捗段階
5. 希望時期・予算帯・意思決定者（概略）
6. 要約 → **利用者修正**
7. 連絡先 + 個人情報同意（**最終ステップのみ PII**）
8. 送信（バックエンドが実行。LLM は送信権限を持たない）

### FAQ 自己解決

- 公開情報で足りる場合は送信せず終了してよい。
- 「解決しましたか」確認後、未解決なら通常フローへ。

### 営業・提案の扱い

完全拒否しない。3 段階:

| 段階 | 扱い |
|---|---|
| 具体性が高い | 担当へ即通知 |
| 判断材料不足 | 日次ダイジェスト等の低優先 |
| 無差別営業 | 隔離キュー |

### 保存

- 生会話全文の無期限保存はしない。要約と必要項目を主記録（corsweb PP / ADR-0006）。

### AI 権限境界

AI（LLM）は分類・要約・質問文の生成に限定。メール送信・社内検索・契約確定はバックエンドの決定論的処理のみ。

## 理由

- B2B 購入判断者の第一印象とリード品質を優先する。
- SNS キャラ資産は破棄せずモード分離で残す。
- 要件定義の「フォームを審査付き受付へ」と一致する。

## 影響

- `translations.ts` / プロンプト / UI のモード分岐。
- intent は corsweb 正本と同一キー。
- 構造化フロー Issue（#6 系）と同期。

## 代替案

- **全画面を福岡弁のまま**: 監査・要件不合格のため却下。
- **常に 10 問以上の深掘り**: 離脱増のため却下（最大 7 問程度）。
