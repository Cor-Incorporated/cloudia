# AI Contact Reception — 要件要約（Cloudia / Cor 正本）

仮称: **AI問い合わせ受付システム（AI Contact Reception）**  
実装ブランド: **Cloudia**  
配置先: **corsweb お問い合わせフォーム代用**（専用 URL / 同一オリジン埋め込み）

本ファイルは長い要件定義書の **Cor・Cloudia 向け要約正本**である。詳細の議論は元文書を参照し、実装判断は本リポ ADR を優先する。

## 1. 目的

従来型お問い合わせフォームを、対話型 AI 受付に置き換える。

| # | 目的 |
|---|---|
| 1 | 迷惑営業・スパム問い合わせの削減 |
| 2 | 日本語以外（初期は英語）への対応 |
| 3 | AI による内容整理・不足情報の補完 |
| 4 | 重要度・緊急度・商談可能性の判定支援 |
| 5 | 担当者が確認すべき案件のみ通知 |
| 6 | 受付後の社内処理効率化 |
| 7 | 分析可能な構造化データの蓄積 |

**AI が完結しないもの**: 契約・見積確定・法的判断・技術保証・採用・支払。これらは人間が最終判断する。

## 2. AI の役割境界

| やってよい | やってはいけない |
|---|---|
| 一次受付・ヒアリング・FAQ | 任意宛先へのメール送信を LLM が直接実行 |
| 分類・部署振り分け提案 | 契約・見積の確定 |
| 要約・構造化 | 社内データ横断検索・無制限通知 |
| 明らかな spam の隔離候補提示 | 完全自動の問い合わせ拒否（誤判定リスク） |

原則: **LLM に権限を与えず**、分類結果を返し、決定論的バックエンドが実行する（ADR-0006）。

## 3. 基本フロー（要件 §6.1）

1. 利用開始（PP / AI 利用注意）
2. BOT 検証（Turnstile 等）
3. 言語判定（ja/en）
4. 問い合わせ目的（intent）
5. 対話ヒアリング（通常 3〜6 往復、必須最大約 7 問）
6. 分類・営業/spam 判定
7. 要約 → **利用者による確認・修正**
8. 連絡先入力（PII はここで）
9. メールアドレス認証（Phase 1 目標）
10. 登録・受付番号・完了通知
11. 担当へ構造化通知（メール / 将来 Slack）

**FAQ のみ解決**: 送信せず終了可。  
**営業疑い**: 営業専用質問 → スコア → 即通知 / 日次 / 隔離の 3 段階。

## 4. Phase 1 コア要件（初期リリース）

- [ ] 日本語・英語
- [ ] 問い合わせ目的（intent）分類
- [ ] 分類別ヒアリング + AI 追加質問
- [ ] FAQ 自己解決
- [ ] 問い合わせ要約 + ユーザー確認
- [ ] 連絡先取得（submit 時のみ PII）
- [ ] メールアドレス認証
- [ ] Cloudflare Turnstile
- [ ] IP・セッション レート制限
- [ ] 営業提案の分離（3 段階）
- [ ] スパムリスクスコア
- [ ] Slack・メール通知（初期はメール必須）
- [ ] AI 障害時の代替フォーム
- [ ] AI 出力のスキーマ検証
- [ ] プロンプトインジェクション対策
- [ ] 個人情報の保存期間・削除方針
- [ ] LLM プロバイダ交換可能な Gateway
- [ ] 簡易管理 or メール Inbox で運用可能であること

## 5. 初期除外（要件 §20）

自動見積確定、契約締結、日程自動確定、自動メール返信、社内機密横断検索、音声通話、顔認証、SNS 連携、外部 Web 自律調査、完全自動拒否、添付の全面 AI 解析。

## 6. 重要な設計判断（要件 §21）

1. **フォーム完全廃止は危険** → AI 主導線 + 最小 fallback（同等の BOT 検証）
2. **spam 撃退は LLM だけに頼らない** → Turnstile / rate limit / メール認証 / ハニーポット / ルール → AI
3. **営業を全拒否しない** → 具体性で 3 段階キュー
